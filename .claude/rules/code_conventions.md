# 📄 Claude Code Python 협업 및 코딩 컨벤션

## 🎯 목적
이 문서는 팀원 및 다른 AI 에이전트들과의 협업을 최우선으로 고려하여 작성되었습니다. 작성된 코드가 읽기 쉽고, 유지보수가 용이하며, 자동화된 검증 도구(Ruff 등)를 매끄럽게 통과할 수 있도록 **고품질 파이썬 코드 구조화 규칙**을 정의합니다.

## 1. 포맷팅 및 린팅 (Ruff & PEP8)
본 프로젝트는 일관된 코드 스타일과 협업 효율을 위해 파이썬 포맷터 및 린터로 **Ruff**를 선호합니다.
* **엄격한 PEP8 준수**: 모든 파이썬 코드는 표준 스타일 가이드인 PEP8을 엄격히 따릅니다.
* **Ruff 자동화 검증**: 코드 작성 후에는 터미널에서 반드시 `uv run ruff format .` 과 `uv run ruff check --fix .`를 실행하여 포맷팅을 일치시키고 식별된 린트 에러를 해결합니다.
* **모듈 임포트 구조**: 임포트는 1) 표준 라이브러리, 2) 서드파티 패키지, 3) 로컬 프로젝트 모듈 순서로 그룹화하여 기재하며 정렬은 Ruff에 위임합니다.

## 2. 코드 퀄리티 및 설계 (협업 지향)

### 2.1 가독성과 단일 책임 (KISS & SRP)
* 코드를 작성하는 시간보다 남이 읽는 시간이 훨씬 길다는 점을 명심하고, 복잡한 숏코딩보다 **단순명료한 가독성**을 최우선합니다.
* 함수와 클래스는 **단일 책임(Single Responsibility)** 원칙을 따릅니다. 하나의 함수가 너무 많은 역할을 짊어진다면 즉시 여러 개의 작은 함수로 분리합니다.
* 네이밍은 동작의 의도를 완벽히 설명해야 합니다. `run`, `process`, `handle`, `data`, `info` 등의 모호한 이름은 엄격히 금지합니다. (예: `process_data()` -> `extract_user_emails()`)

### 2.2 방어적 프로그래밍 (Defensive Programming)
* **타입 힌팅 (Type Hinting) 100%**: 모든 함수 시그니처와 클래스 변수에는 인자와 리턴 타입 힌트를 반드시 명시해야 합니다. `Any` 타입의 사용은 설계의 실패로 간주하고 극도로 지양합니다.
* **명시적 예외 처리**: `except Exception:`과 같은 광범위한 강제 예외 묵살은 에러 발생 원인 추적을 치명적으로 방해하므로 금지합니다. 파악 가능한 `ValueError`, `KeyError` 등 구체적이고 좁은 범위의 예외 상황만을 핸들링해야 합니다.

### 2.3 매직 넘버 및 매직 스트링의 배제
* 코드 중간에 하드코딩된 숫자나 의미를 알 수 없는 문자열이 존재하면 유지보수성이 급감합니다.
* 의미를 지니는 상수 값들은 파일 상단이나 별도의 상수/설정 파일에 `UPPER_SNAKE_CASE` 상수로 추출하거나, 상태 등은 `enum.Enum`을 활용해 반드시 열거형 객체로 관리합니다.

### 2.4 디버깅과 로깅 (Logging vs Print)
* 디버깅이나 상태 추적 목적으로 `print()` 함수를 흩뿌려 두는 것을 엄격히 금지합니다. 이는 프로덕션 환경의 표준 출력을 오염시키며, 추후 관리가 불가능합니다.
* 반드시 파이썬 내장 `logging` 모듈(또는 `loguru` 등)을 사용하여, 시스템 상태나 변수값 확인은 `logger.debug()`나 `logger.info()`를 활용하세요. 이를 통해 배포 시 로그 레벨만 조절하여 디버그 메시지를 손쉽게 숨길 수 있어야 합니다.

## 3. Docstring 규칙 (Google Style)
모든 주요 모듈, 클래스, 함수(메서드)에는 **Google Style Docstring**을 필수로 작성하여 협업 시 IDE 내에서 즉각적인 도움을 주어야 합니다.
* **목적과 Context**: 함수가 왜 존재하는지, 어떤 영향을 끼치는지를 첫 줄에 명확히 작성합니다.
* `Args:`, `Returns:`, `Raises:` (예외 발생 가능성이 있을 시) 항목을 반드시 포함합니다.
* 복잡하거나 헷갈리기 쉬운 로직의 경우 `Example:` 항목을 활용하여 사용법을 설명합니다.

```python
def search_products(query: str, max_results: int = 10) -> list[dict]:
    """
    사용자의 자연어 요청을 기반으로 연관성 높은 상품을 검색한다.

    Args:
        query (str): 사용자가 입력한 자연어 검색어 (예: "노이즈 캔슬링 헤드폰")
        max_results (int, optional): 반환할 최대 검색 결과 수. 기본값은 10.

    Returns:
        list[dict]: 검색된 상품 딕셔너리의 리스트

    Raises:
        ConnectionError: 검색 서버와 연결할 수 없을 때 발생
    """
```

## 4. 구조화된 데이터 계층 (Pydantic & Annotated)

복잡한 입력 파라미터나 API 데이터 교환 및 설정 객체에는 단순 딕셔너리(`dict`) 사용을 피하고 무조건 **Pydantic** 모델을 배정합니다.
* **자동 검증 (Validation)**: 각 필드에 제약 조건(`Field`)을 걸어 도메인 로직에 잘못된 유형이나 범위의 데이터가 유입되는 것을 원천 차단합니다.
* **의미 부여 (Annotated)**: 파라미터가 무슨 역할을 하는지 `typing.Annotated`를 사용해 의미와 예시를 명확히 작성하여 다른 개발자나 AI가 쉽게 이해할 수 있게 돕습니다.

```python
from pydantic import BaseModel, Field
from typing import Annotated

class SearchInput(BaseModel):
    query: Annotated[str, "상품 설명을 나타내는 자연어 검색어"]
    max_price: Annotated[int, "최대 검색 가격(원)"] = Field(gt=0, le=1000000)
```

## 5. 기타 필수 코딩 규칙
* **f-string 강제**: 문자열 병합이나 포매팅에는 반드시 `f-string` 기법만을 사용합니다. (`.format()`이나 `%` 방식 금지)
* **Early Return 권장**: 중첩 뎁스(Indent)가 깊어지면 가독성이 현저히 떨어집니다. 조건부 가드(Guard Clauses) 구문을 통해 부정 조건이나 예외 상황을 메서드 상단에서 바로 `return` 하거나 `raise` 하는 방식(Early Return)을 적극 도입합니다.
* **한글 주석 원칙 (Why)**: "무엇을(What)" 하는지 설명하는 설명충식 주석은 피해야 합니다(코드로 말해야 함). 비즈니스 결정, 우회해야 했던 시스템 한계 등 **"왜 이렇게 했는지(Why)"**에 국한된 정보만 한글 주석으로 간결하게 기재합니다.
