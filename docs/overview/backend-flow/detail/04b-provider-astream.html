Provider Streaming

④b provider.astream() — LLM 스트리밍 어댑터

_run_agent_turn 루프의 ③ LLM 스트리밍 노드 한 칸의 속살. 이 메서드가 무슨 인자를 받고 무엇을 yield 하기에 루프가 한 라운드를 돌고 멈출 곳에서 멈추는지를 — 흐름도 없이 — 계약 단위로 설명한다.

1

계약 한눈에 — LLMProvider Protocol

harness 는 어떤 LLM 을 쓰는지 모른다. providers/factory.py 의 LLMProvider 는 메서드 하나짜리 Protocol 이고, 루프는 이 한 메서드의 이벤트 계약에만 의존한다. 그래서 OpenAI·Mock·향후 신규 provider 가 자유롭게 갈아끼워진다.

class LLMProvider(Protocol): async def astream(self, messages, tools) -> AsyncIterator[StreamEvent]: # delta / reasoning / tool_call / done 이벤트를 순서대로 yield ...
hot-swap: /api/chat 요청마다 get_provider(settings) 가 최신 설정으로 인스턴스를 즉석 생성한다(factory.py). 서버 재시작 없이 provider·모델 전환이 즉시 적용되는 이유 — 루프는 매번 "astream 을 가진 무언가"만 건네받는다.
2

입력 인자 — messages · tools

인자타입무엇 / 어디서
messages list[Message] 이번 라운드에 LLM 이 볼 전체 컨텍스트 — system·user·assistant·tool role (agent/models.py Message). 루프가 in-place 로 누적하는 호출자 소유 리스트. OpenAI 어댑터는 이를 wire 포맷으로 변환 (openai.py · _convert_message_to_wire; tool 메시지는 tool_call_id 로 매칭).
tools list[ToolSpec] 이번 턴 LLM 에 노출할 도구 JSON Schema 목록 (models.py ToolSpec(name·description·parameters)). 호출부에 따라 specs·tool_specs·sub_specs 로 불리는 같은 인자. 빈 리스트면 도구 없이 텍스트만 생성하게 한다.

호출부는 두 곳뿐이다 (둘 다 harness/loop.py):

# 일반 라운드 — 선별된 도구 스펙을 노출 async for event in provider.astream(messages, sub_specs): ... # max_iterations fallback(F6) — 도구 없는 최종 요약 라운드 async for event in ctx.provider.astream(ctx.messages, []): ...
3

yield 하는 이벤트 — 4종

provider 가 실제로 내보내는 이벤트는 네 가지다(모두 StreamEvent 유니온, models.py). 앞 셋은 라운드 도중, 마지막 하나가 라운드를 닫는다.

이벤트type의미 · 타이밍
DeltaEventdelta응답 본문 토큰 조각 — 도착 즉시 yield(프론트 타이핑).
ReasoningEventreasoningo-series 모델의 thinking 토큰(delta.reasoning_content) — 없는 모델이면 안 나온다.
ToolCallEventtool_call완성된 도구 호출 1건. 스트리밍 조각을 index별 버퍼에 누적했다가 스트림 끝에서 flush(openai.py).
DoneEventdone이 라운드 종료 신호 — 정확히 1회 보장. 스트림이 어떻게 끝나든 마지막에 한 번 나온다.
순서 규약: delta·reasoning 은 흐르는 대로, tool_call 은 인자 JSON 조각이 다 모인 뒤, DoneEvent 는 맨 마지막. 그래서 done 이 도착한 시점엔 이번 라운드의 모든 tool_call 이 이미 버퍼에 들어와 있다 — 루프가 안심하고 분기할 수 있는 근거.
4

왜 이 이벤트들이 루프를 돌리는가

루프 본문은 astream 을 async for 로 소비하며 event.type 으로만 갈린다 (harness/loop.py · _run_agent_turn):

async for event in provider.astream(messages, sub_specs): match event.type: case "delta": assistant_buffer.append(event.content); yield event case "tool_call": pending_tool_calls.append(event.call); yield event case "reasoning" | "skill_active": yield event # 제어 무관 passthrough case "done": break # ← 이 라운드 종료 case _: yield event; return # 방어적 catch-all

두 가지가 제어 신호다. 나머지는 화면용이다.

신호루프에 미치는 효과
DoneEvent → break 스트리밍 제너레이터의 async for 를 끝내는 유일한 신호. 없으면 라운드가 영영 안 끝난다. provider 가 정확히 1회 보장(F2)하므로 멈춤(hang)도, 중복 break 도 없다.
tool_call 유무 = 분기 없으면 → assistant 텍스트를 모아 append 하고 return(턴 정상 종료). 있으면 → tool_calls 를 실은 assistant 메시지를 append 한 뒤 각 호출을 _handle_tool_call 로 처리하고 다음 iteration 으로. 즉 LLM 은 tool_call 을 emit 함으로써 "도구를 쓰겠다"를 루프에 알린다 → 다음 라운드 발생.
delta / reasoning 제어에 영향 없는 UI passthrough. 단 delta 는 assistant_buffer 에도 모아 히스토리에 남길 assistant content 를 재구성한다.
핵심은 이벤트 타입이 곧 제어 흐름이라는 점이다. 루프는 provider 의 내부 동작을 전혀 모른 채, "done 이 오면 한 라운드 끝", "tool_call 이 끼어 있으면 도구 실행 후 또 돈다" 두 규칙만으로 멀티스텝 에이전트를 돌린다. → 다음: 04 턴 루프 의 전체 §2 루프.
5

스트림 경계 — 복원력 (F2 · F3 · F4)

실 LLM 의 비결정적 스트림(잘림·연결종료·일시 오류)이 루프 불변식을 깨지 않도록 어댑터가 흡수한다.

장치경계처리
F4스트림 생성 단계 일시 오류connection·timeout·rate-limit·5xx 를 지수 백오프로 재시도(최대 2회). 인증·400 등 영구 오류는 즉시 전파. 아무 이벤트도 yield 하기 전이라 재시도 안전(openai.py · _create_stream_with_retry).
F2응답 잘림(finish_reason="length")·연결 종료스트림이 어떻게 끝나든 버퍼된 tool_call 을 flush 하고 DoneEvent 를 정확히 1회 낸다 → 루프가 멈추지 않는다.
F3tool_call 인자 JSON 파싱 실패{} 로 뭉개지 않고 원본을 MALFORMED_TOOL_ARGS_KEY(models.py) 에 보존 → harness 가 "슬롯 누락"으로 오인해 사용자에게 되묻는 대신 LLM 에 재전송을 요구.
이 불변식들이 깨지면 루프가 hang 하거나(done 누락), 멀쩡한 도구 호출이 증발하거나(flush 누락), 엉뚱하게 사용자에게 되묻는다(F3). 자세한 복원력 카탈로그 → 12 안전장치.
Mock 도 같은 계약: MockProvider.astream 은 LLM 없이 동일한 4종 이벤트만 yield 해 시나리오 A~H 를 결정론적으로 재현한다. 도구 실행·tool_result·트레일 등 나머지 UI 이벤트는 전부 harness 가 합성하므로, 같은 루프가 Mock 과 실 LLM 을 구분 없이 돌린다.
